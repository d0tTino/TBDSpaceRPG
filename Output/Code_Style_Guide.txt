# TBD Space Game - Code Style Guide
Version: 1.1
Date: May 1, 2025

## 1. Naming Conventions

### 1.1 General Rules
- Use meaningful and descriptive names
- Avoid abbreviations unless widely known
- Keep names concise but clear
- Use consistent casing rules

### 1.2 C# Naming Patterns

#### 1.2.1 Classes and Types
```csharp
// PascalCase for class names
public class ShipCustomizationManager
{
    // Implementation
}

// Interface names start with 'I'
public interface IUpgradeable
{
    void ApplyUpgrade(UpgradeType type);
}

// Enum names are singular
public enum UpgradeType
{
    Engine,
    Shield,
    Weapon
}
```

#### 1.2.2 Methods and Properties
```csharp
// PascalCase for public methods
public void ApplyUpgrade(UpgradeType type)
{
    // Implementation
}

// Properties use PascalCase
public float ThrustPower { get; private set; }

// Event handlers use EventHandler suffix
private void ShieldDamageEventHandler(object sender, EventArgs e)
{
    // Implementation
}
```

#### 1.2.3 Fields and Variables
```csharp
// Private fields use _camelCase
private float _currentPower;
private ShipSystem _mainEngine;

// Constants use UPPER_CASE
private const float MAX_THRUST = 1000f;
private const int UPGRADE_LIMIT = 5;

// Local variables use camelCase
float currentVelocity = CalculateVelocity();
var shipConfig = new ShipConfiguration();
```

### 1.3 MCP-Specific Naming

#### 1.3.1 Menu Paths
```csharp
// Category/Subcategory/Action format
"MCP/Ship/Create"
"MCP/Ship/V2/Upgrade"
"MCP/Debug/Log"

// Version-specific paths include version
"MCP/Ship/V2/ApplyUpgrade"
"MCP/Crew/V1/Assign"
```

#### 1.3.2 Command Parameters
```csharp
// Parameter names use camelCase
public class McpCommand
{
    public string menuPath;
    public Dictionary<string, object> parameters;
    public Action<bool, string> callback;
}

// Example usage
mcpClient.SendCommand(new McpCommand
{
    menuPath = "MCP/Ship/V2/Upgrade",
    parameters = new Dictionary<string, object>
    {
        { "componentType", "engine" },
        { "upgradeLevel", 2 }
    }
});
```

## 4. Best Practices

### 4.1 Performance Guidelines
```csharp
// Optimize for frequent calls
public float CalculateUpgradeEffect(UpgradeType type)
{
    // Cache frequently accessed values
    if (_cachedEffects.TryGetValue(type, out float effect))
    {
        return effect;
    }

    // Calculate and cache
    effect = ComputeEffect(type);
    _cachedEffects[type] = effect;
    return effect;
}
```

### 4.2 Error Handling
```csharp
// Proper exception handling for AI-generated code
public void ProcessAiGeneratedContent(string content)
{
    try
    {
        ValidateContent(content);
        ApplyContent(content);
    }
    catch (ContentValidationException e)
    {
        Debug.LogError($"AI content validation failed: {e.Message}");
        TriggerFallbackBehavior();
    }
    catch (Exception e)
    {
        Debug.LogError($"Unexpected error in AI content: {e.Message}");
        throw;
    }
}
```

### 4.3 Testing Requirements

## 5. Automated Formatting
The repository includes an `.editorconfig` file that defines consistent C#
styling rules such as 4-space indentation and groupings of `using`
directives.  Continuous integration runs `dotnet format --verify-no-changes`
against the `OrbitalMechanics.sln` solution to ensure all committed code
adheres to these settings.
