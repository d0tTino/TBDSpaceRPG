1.0 Development Environment Setup
================================
This section details the setup process for the development environment, including required software, tools, and configuration steps.

1.1 Required Software and Tools
-----------------------------
- Unity 2025.1 LTS or newer (project is built on 2025.1.5f1 specifically)
- Visual Studio 2025 or JetBrains Rider 2025.1+
- Git 2.40.0+ for version control
- Git LFS 3.4.0+ for large file storage
- PowerShell 7.5 or newer for automation scripts
- tmux 3.4+ for terminal session management
- Python 3.11+ for auxiliary tools and AI integration scripts
- Bun 1.1.0+ for JavaScript runtime and package management
- Unity Test Framework 2.0+ for unit testing
- DOTween Pro 2.0+ for animation
- Cinemachine 3.0+ for camera systems
- TextMeshPro 4.0+ for UI text rendering
- Rust 1.72+ for performance-critical components
- Docker 25.0+ for containerized build environments

1.2 Hardware Requirements
-----------------------
Development hardware should meet or exceed:
- CPU: 8+ cores, 3.5GHz+
- RAM: 32GB minimum, 64GB recommended
- GPU: NVIDIA RTX 4070 / AMD RX 7700 or better 
- Storage: 500GB+ NVMe SSD for project files
- Display: 1440p or higher resolution, dual monitor setup recommended
- Network: 200Mbps+ connection for asset syncing

2.0 Version Control Guidelines
=============================
The TBD Space Game project uses Git for version control with a standardized workflow to ensure code quality and team coordination.

2.1 Repository Structure
-----------------------
The main repository is organized as follows:
- `/Assets` - Unity project assets
- `/Documentation` - Project documentation
- `/Tools` - Development and build tools
- `/Tests` - Automated tests

2.2 Branching Strategy
--------------------
The project follows a modified GitFlow branching strategy:

- `main` - Stable production-ready code
- `develop` - Integration branch for features
- `feature/<feature-name>` - Feature development branches
- `bugfix/<bug-id>` - Bug fix branches
- `release/<version>` - Release preparation branches
- `hotfix/<hotfix-id>` - Critical production fixes

2.3 Commit Message Standards
--------------------------
All commit messages must follow this format:
```
[Area]: Brief description of change (50 chars max)

Detailed explanation of what this change does and why.
Include any relevant background information or context.
Wrap lines at 72 characters.
```

Areas include:
- `[Core]` - Core game systems
- `[UI]` - User interface changes
- `[Crew]` - Crew system modifications
- `[Ship]` - Ship/habitat systems
- `[Tech]` - Technical implementation
- `[Docs]` - Documentation updates
- `[Build]` - Build system changes
- `[Test]` - Test-related changes

2.4 Pull Request Process
----------------------
1. Create a pull request from your feature branch to develop
2. Ensure all tests pass and CI checks succeed
3. Obtain code review from at least one team member
4. Address all review comments
5. Squash commits if necessary for clarity
6. Maintainer merges after approval

2.5 Versioning
------------
The project follows Semantic Versioning (SemVer):
- MAJOR.MINOR.PATCH (e.g., 1.2.3)
- Increment MAJOR for incompatible API changes
- Increment MINOR for backward-compatible functionality
- Increment PATCH for backward-compatible bug fixes

2.6 Documentation Versioning
--------------------------
Documentation files should:
- Include version date at the top (e.g., "Version: April 2025")
- Be updated whenever related code changes
- Maintain a changelog section for major updates
- Follow the same branch strategy as code

2.7 Large File Storage
-------------------
Use Git LFS for:
- Texture files (.psd, .tga, .png, .jpg) over 5MB
- 3D models (.fbx, .obj)
- Audio files (.wav, .mp3)
- Video files (.mp4, .mov)
- Unity asset bundles

3.0 Coding Standards
===================
C# Coding Standards:
- Use PascalCase for public members, camelCase for private members, and _camelCase for private fields
- Include XML documentation for all public methods and classes
- Follow MonoBehaviour lifecycle patterns and prefer composition over inheritance
- Consider performance implications, especially for methods called frequently

Code Structure and Organization:
- All gameplay logic belongs in Scripts/ directory
- Manager classes should be in Scripts/Managers/
- Data models should be in Scripts/Models/
- UI components should be in Scripts/UI/
- Implement event-based communication between systems
- Each class should have a single responsibility
- Prefer composition over inheritance
- Keep methods short and focused on a single task

Naming Conventions:
- Class names should be nouns and use PascalCase
- Method names should be verbs and use PascalCase
- Interface names should start with 'I' and use PascalCase
- Private fields should be prefixed with underscore and use camelCase
- Constants should use ALL_CAPS with underscores
- Boolean variables should start with 'is', 'has', 'can', etc.

5.2 Unit Testing Guidelines
------------------------
Unit tests should:
- Be located in Scripts/Tests/EditMode/
- Follow AAA pattern (Arrange-Act-Assert)
- Mock dependencies using NSubstitute
- Focus on single unit of functionality
- Run quickly (<100ms per test)
- Have clear naming: `[Method]_[Scenario]_[ExpectedResult]`

Example unit test:
```csharp
[Test]
public void CalculateOrbit_WithValidParameters_ReturnsExpectedPath()
{
    // Arrange
    var calculator = new OrbitCalculator();
    var parameters = new OrbitParameters(mass: 1000, velocity: new Vector3(0, 0, 10));
    
    // Act
    OrbitPath result = calculator.CalculateOrbit(parameters);
    
    // Assert
    Assert.IsNotNull(result);
    Assert.AreEqual(expectedCount, result.points.Count);
    Assert.IsTrue(MathUtils.IsWithinTolerance(expectedPeriod, result.period, 0.001f));
}
```

6.0 MCP Integration Workflow
===========================
This section documents the workflow for integrating with the Machine Control Protocol (MCP) system, which manages hardware control interfaces for the game.

6.1 MCP Overview
--------------
The Machine Control Protocol allows the game to interface with:
- Custom hardware controllers
- Multi-display setups
- IoT devices for immersive feedback
- Custom input peripherals
- Simulator hardware platforms

This integration requires special care due to hardware dependencies and safety considerations.

6.2 MCP Architecture
-----------------
The MCP system consists of:
- Core MCP library (`/Assets/Scripts/MCP/Core/`)
- Device adapters (`/Assets/Scripts/MCP/Adapters/`)
- Abstraction layer (`/Assets/Scripts/MCP/Abstraction/`)
- Hardware simulators for development (`/Assets/Scripts/MCP/Simulators/`)
- Safety monitoring services (`/Assets/Scripts/MCP/Safety/`)

6.3 Development Environment Setup for MCP
--------------------------------------
To develop with MCP features:

1. Install required hardware drivers:
   - XInput controllers driver
   - Custom hardware interface drivers (if available)
   - MCP device simulator package

2. Configure hardware access permissions:
   - Configure device permissions on development machine
   - Set up MCP simulation mode for non-hardware testing

3. Enable MCP features in Unity:
   ```csharp
   // In a startup script or manager
   MCPSystem.Initialize(new MCPConfiguration 
   { 
       EnableHardwareAccess = true,
       SimulateUnavailableDevices = true,
       LogLevel = MCPLogLevel.Verbose
   });
   ```

6.4 MCP Development Workflow
-------------------------
1. Develop against the abstraction layer:
   ```csharp
   // Instead of direct hardware access:
   // DON'T: serialPort.Write(command);
   
   // DO: Use the abstraction layer
   IMCPDevice device = MCPSystem.GetDevice("MainControlPanel");
   device.SendCommand(new ControlCommand("SetLED", 2, true));
   ```

2. Test using hardware simulators:
   ```csharp
   // Enable simulator mode
   MCPSystem.SetSimulationMode(true);
   
   // Run standard hardware interactions
   // Check simulator visual feedback
   ```

3. Create device adapter tests:
   ```csharp
   [Test]
   public void DeviceAdapter_SendingValidCommand_ReturnsSuccess()
   {
       var adapter = new MockControlPanelAdapter();
       MCPResponse response = adapter.ProcessCommand(new ControlCommand("SetLED", 1, true));
       Assert.AreEqual(MCPResponseStatus.Success, response.Status);
   }
   ```

4. Hardware integration testing:
   - Run regular integration tests with simulator  
   - Document which tests require physical hardware
   - Mark hardware-dependent tests with category attribute
   - Use tmux to manage multiple terminal sessions for hardware testing

6.5 MCP Safety Guidelines
----------------------
When working with physical hardware:

1. All direct hardware commands must use safety checking:
   ```csharp
   // Add safety boundaries
   if (!SafetyChecker.IsCommandSafe(command, device))
   {
       Debug.LogError("Unsafe command detected!");
       return MCPResponse.Unsafe;
   }
   ```

2. Include hardware failure handling:
   ```csharp
   try 
   {
       device.SendCommand(command);
   }
   catch (MCPHardwareException ex)
   {
       // Log failure
       MCPSystem.LogFailure(device.DeviceId, ex);
       
       // Fallback to safe state
       device.ResetToSafeState();
       
       // Notify game systems
       EventBus.Publish(new HardwareFailureEvent(device.DeviceId, ex.FailureType));
   }
   ```

3. All hardware-interacting code undergoes special review

6.6 MCP Version Compatibility
--------------------------
MCP integration requires version compatibility management:

1. MCP library versioning follows SemVer
2. Hardware firmware versions must be compatible
3. Version compatibility matrix in `/Assets/Scripts/MCP/VersionCompatibility.json`
4. Version checking at runtime:
   ```csharp
   if (!MCPSystem.IsCompatibleWithHardware(hardwareVersion))
   {
       Debug.LogError($"Incompatible hardware version: {hardwareVersion}");
       // Handle incompatibility
   }
   ```

6.7 MCP Documentation Requirements
-------------------------------
For MCP-related features:

1. Document all hardware dependencies
2. Include circuit diagrams where applicable
3. Specify required firmware versions
4. Document safety considerations and limits
5. Include simulation mode instructions
6. Add hardware troubleshooting guide

8.0 Issue Tracking and Resolution
================================
This section details the procedures for tracking, prioritizing, and resolving issues throughout the development lifecycle.

8.1 Issue Tracking System
----------------------
The project uses GitLab Issues for issue tracking with the following structure:

- Projects organized by system area
- Issues categorized by type, severity, and priority
- Automated linking between issues and commits
- Milestone-based planning and tracking
- Time estimation and tracking
- Custom workflow with project-specific states

8.2 Issue Types and Categories
---------------------------
Issues are categorized as:

| Type | Description | Label | Resolution Time Target |
|------|------------|-------|------------------------|
| Bug | Functionality not working as expected | `bug` | Based on severity |
| Feature | New functionality to be added | `feature` | Based on planning |
| Task | Development work item | `task` | Based on estimate |
| Improvement | Enhancement to existing functionality | `improvement` | Based on priority |
| Documentation | Documentation updates | `documentation` | 1-2 days |
| Technical Debt | Code cleanup or refactoring | `tech-debt` | By sprint allocation |

Bug severity levels:
- **Critical**: Game-breaking, blocks progress (24-hour response)
- **Major**: Significant feature broken (3-day response)
- **Normal**: Feature works incorrectly (next sprint)
- **Minor**: Cosmetic or edge case (backlog)

8.3 Issue Reporting Process
------------------------
When reporting an issue:

1. Check if the issue already exists
2. Create a new issue with:
   - Clear descriptive title
   - Detailed reproduction steps
   - Expected vs. actual behavior
   - Screenshots or videos if applicable
   - System information if relevant
   - Version information

3. Apply appropriate labels:
   - Type label (`bug`, `feature`, etc.)
   - Area label (`ui`, `combat`, `crew`, etc.)
   - Priority label if known

4. For bugs, include:
   - Saved game file if applicable
   - Log files
   - Steps to reproduce consistently
   - Impact/severity assessment

8.4 Triage and Prioritization
--------------------------
Issues are triaged:

1. Daily triage meeting for new issues
2. Severity assessment by lead developer
3. Priority assignment based on:
   - Severity (for bugs)
   - Strategic importance (for features)
   - Dependencies with other work
   - Resource availability

4. Assignment to milestone or sprint
5. Developer assignment or self-selection from pool

8.5 Issue Resolution Workflow
--------------------------
When resolving an issue:

1. Assign issue to yourself
2. Create branch from develop:
   ```
   git checkout -b bugfix/issue-123 develop
   ```
   or
   ```
   git checkout -b feature/issue-456 develop
   ```

3. Implement fix or feature
4. Add tests to verify the fix
5. Ensure all tests pass
6. Submit pull request referencing the issue:
   - Include "Fixes #123" in PR description
   - Attach screenshots of fixed behavior if applicable
   - Complete the PR template with all required information

7. Address review feedback
8. After approval and merge:
   - Verify fix in develop branch
   - Update issue with any additional information
   - Close issue if completely resolved

8.6 Regression Testing
-------------------
For bug fixes:

1. Create a specific test to verify the fix
2. Add to regression test suite
3. Document the root cause in the issue
4. Add prevention notes for similar issues

8.7 Bug Metrics and Analysis
-------------------------
The team tracks:

1. Bug discovery rate per sprint
2. Average time to resolution
3. Regression rate (fixed bugs that reappear)
4. Bug distribution by system area
5. Trends in bug categories

Monthly bug review meetings analyze:
- Common root causes
- Areas with high bug density
- Developer education needs
- Process improvement opportunities

8.8 Feature Requests and Enhancements
----------------------------------
For new features and enhancements:

1. Submit detailed proposal with rationale
2. Include mockups or prototypes if applicable
3. Product owner reviews and prioritizes
4. Approved features are scheduled into roadmap
5. Large features broken into smaller implementable tasks
6. Implementation follows standard development workflow

8.9 Knowledge Base Integration
---------------------------
Issues feed into the knowledge base:

1. Common issues documented in Wiki
